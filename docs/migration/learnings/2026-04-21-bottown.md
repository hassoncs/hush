## Blockers encountered

### Blocker 1: `.pnpm-store` at project root caused ENOENT in migrate scanner

The project formerly lived at `platform/bottown` and had its pnpm store at that path. After moving to `platform/opencode-ecosystem/bottown`, a stale `.pnpm-store/` directory was left at the project root. The hush `collectCommandReferences` scanner traverses all subdirectories (excluding only `node_modules`, `.git`, `.hush`) and failed with:

```
Error: ENOENT: no such file or directory, stat '/Users/hassoncs/.../bottown/.pnpm-store/v10/projects/29b549279af7941a49424f275fb9d7f0'
```

The pnpm store `index/` had entries pointing to a `projects/` path that no longer contained the expected symlink entries.

**Workaround:** Temporarily renamed `.pnpm-store` → `.pnpm-store-backup`, ran migration, renamed back.

### Blocker 2: Scanner hangs on 2000+ markdown files

The bottown project has ~2125 markdown files outside `node_modules`. The `collectCommandReferences` scanner reads every `.md`/`.mdx`/`package.json`/`README` file to find `hush ` command references. The dry-run timed out (killed at 143) scanning the full tree.

Large directories by markdown count:
- `groups/`: 1742 markdown files (gitignored agent session logs)
- `.pnpm-storybook-store/`: 1728 markdown files (package README files)
- `container/`: 776 markdown files

**Workaround:** Temporarily renamed `groups/`, `.pnpm-storybook-store/`, `container/` before running real migrate, restored afterward.

**Note:** Dry-run was skipped entirely due to scanner timeout. Real migration was run directly and succeeded.

## Project-specific quirks

- **Stale `.npmrc` store-dir**: Points to `platform/bottown/.pnpm-store` (old project path). Not corrected in this migration — unrelated to hush, but worth noting if pnpm install issues arise.
- **`.hush.encrypted.local` naming mismatch**: The local override file uses `.hush.encrypted.local` naming instead of `.hush.local.encrypted`. hush migrate v2 source inventory expects the latter. The local file is an orphan with the same 21 secrets as shared. No data loss since all values were captured in the shared source.
- **Key discovery requires SOPS_AGE_KEY_FILE env**: Without it, `hush status` prints the v3 header but then fails decryption. Fixed via `.envrc`. The root cause is that `package.json` has no `repository` field, so `getProjectIdentifier()` returns undefined, and `loadV3Repository()` can't bootstrap the identity to read `metadata.project` from the manifest.

## Hush defects found

### Defect: `collectCommandReferences` scanner does not skip `.pnpm-store*` directories

**Severity:** Medium — blocks migration entirely for projects with stale pnpm stores at project root.

**Repro steps:**
1. Project has a `.pnpm-store/` directory at root (e.g., moved from old location)
2. pnpm `index/` entries reference symlinks in `projects/` that no longer exist
3. Run: `hush migrate --from v2 --dry-run`

**Exact error:**
```
Error: ENOENT: no such file or directory, stat '/path/to/.pnpm-store/v10/projects/<hash>'
```

**Expected behavior:** Scanner should skip `.pnpm-store*`, `.pnpm-storybook-store`, and similar pnpm cache directories (analogous to skipping `node_modules`).

**Actual behavior:** Crashes with ENOENT when traversing stale symlink targets.

---

### Defect: `collectCommandReferences` scanner hangs on repos with 2000+ markdown files

**Severity:** Medium — dry-run unusable on large repos; real migration requires workaround.

**Repro steps:**
1. Project has 2000+ `.md`/`.mdx` files outside `node_modules` (e.g., agent session logs in `groups/`, storybook package READMEs in `.pnpm-storybook-store/`)
2. Run: `hush migrate --from v2 --dry-run`

**Observed:** Command runs for >2 minutes without output and is killed (exit 143).

**Expected behavior:** Scanner should have a timeout or skip directories that are gitignored.

**Actual behavior:** No timeout, no progress indicator, hangs indefinitely.

**Note:** The scan result only populates `legacyMigration.references` metadata in the manifest — it is informational only. A future fix could make this scan optional or background.

---

### Defect: `hush status` prints output twice when key resolution fails then falls back

**Severity:** Low — cosmetic confusion.

**Repro:** Run `hush status` without `SOPS_AGE_KEY_FILE` set in a v3 repo where the key cannot be auto-discovered. Status prints `Repository: v3` header then immediately re-runs and prints `Repository: missing` with the SOPS decryption failure.

**Expected:** Single status output; if decryption fails, show the error once.
